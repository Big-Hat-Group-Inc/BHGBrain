## Background

BHGBrain uses Qdrant Cloud as a shared vector store across multiple devices. Each device has its own local SQLite as a metadata index. On a new device, SQLite is empty. Qdrant already had 141 vectors from Device 1. BHGBrain health showed memory_count: 0 and all searches returned empty — because the search path JOINs Qdrant results against SQLite rows, and there are none on a new device. No bootstrap/sync mechanism exists.

## Task 1: Create GitHub issue

Run this exact gh command:
gh issue create --repo Big-Hat-Group-Inc/BHGBrain --title "feat: new-device bootstrap - hydrate SQLite from Qdrant on first run" --body "When BHGBrain is installed on a new device, local SQLite is empty. Qdrant Cloud already contains memories from other devices, but all searches return empty because the search path JOINs Qdrant results against local SQLite rows which do not exist on the new device. Observed: Device 2 started fresh, Qdrant had 141 vectors from Device 1, memory_count showed 0, all searches empty. Proposed fix: on startup when memory_count=0 and Qdrant has data, scan all bhgbrain_* collections, reconstruct MemoryRecord rows from Qdrant payloads, upsert into SQLite. Also add bhgbrain repair --from-qdrant CLI command."
